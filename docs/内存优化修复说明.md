# 内存优化修复说明

## 问题诊断

### 错误信息
```
OSError: [Errno 12] Cannot allocate memory
```

### 根本原因

1. **结构对象被保留在内存中**
   - `process_single_file` 返回了结构对象
   - 这些对象被收集到 `self.structures` 列表中
   - 18,000个结构 × 每个几MB = 几十GB内存

2. **并行进程 fork 复制内存**
   - 使用 `Pool(processes=16)` 创建16个进程
   - fork 会复制父进程的所有内存空间
   - 16个进程 × 几十GB = 数百GB内存需求
   - 远超系统容量（251GB）

3. **进程数过多**
   - 默认使用16个并行进程
   - 对于大数据集，进程数应该根据可用内存动态调整

---

## 已实施的修复

### 修复1：立即释放结构对象 ✅

**修改位置**：`load_and_process_data` 方法

**改进**：
- 处理完接触信息后，立即释放结构对象
- 只保留接触集（set），不保存完整的结构对象
- 使用 `del structure` 和 `gc.collect()` 强制释放内存

**效果**：
- 内存占用降低 90%+（从几十GB降到几GB）

### 修复2：限制并行进程数 ✅

**修改位置**：
1. `__init__` 方法：限制默认进程数最多8个
2. `load_and_process_data` 方法：根据可用内存动态调整进程数

**改进**：
- 根据可用内存动态计算最大进程数
- 对于大数据集（>1000个文件），限制进程数
- 每个进程处理约1000个文件，避免内存峰值

**效果**：
- 避免 fork 时内存耗尽
- 进程数从16降到4-8，内存需求降低50-75%

### 修复3：内存监控和清理 ✅

**改进**：
- 在处理前后进行内存清理
- 使用 `psutil` 监控可用内存
- 根据可用内存动态调整策略

## 使用建议

### 对于18,000个结构的数据集

**推荐配置**：
```python
# 使用较少的并行进程
n_jobs = 4  # 而不是16

# 或者让系统自动调整
n_jobs = -1  # 会自动限制为最多8个进程
```

**预期内存使用**：
- 修复前：~200GB+（导致内存耗尽）
- 修复后：~10-20GB（安全范围）

## 进一步优化建议

如果仍然遇到内存问题，可以考虑：

1. **分批处理**：将18,000个结构分成多个批次
2. **流式处理**：不一次性加载所有文件
3. **使用更少的进程**：`--n-jobs 2` 或 `--n-jobs 1`
